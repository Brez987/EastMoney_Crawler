# 000002 爬虫数据质量复核报告

## 结论

1. `data/000002_enhanced.csv` 是基于项目原数据 `数据/000002.csv` 做增量补爬后的产物；当前流水线不会回头补爬原数据已有日期之前的历史页。
2. 因此，爬虫产物与外部论文数据在 2008-2018 年几乎不重合，主要说明 **项目原数据源 `数据/000002.csv` 的历史覆盖不如外部数据集**。
3. 从新增数据本身看，本次补爬新增 `115,279` 条，全部晚于项目原数据最大时间 `2025-02-06 20:56:24`，且新增 `post_id` 无重复，核心列表字段大多完整，说明增量补爬主流程是有效的。
4. 但新增数据整体还不能完全比肩外部论文数据：新增记录中 `24,065` 条 `post_type` 与 `user_id` 为空，占 `20.88%`；`92` 条 `click_count` 为 `1.0万`、`3.9万` 这类非纯数字；`3,765` 条 `content` 为空，占 `3.27%`，其中财富号正文空值尤其明显。

最终建议：**可以考虑把 `outsource_check/000002.csv` 转换为新的历史基础数据源，再用当前爬虫流水线从外部数据截止日期之后继续补爬。** 但在批量推广前，还应修复新增数据中的 `post_type/user_id` 缺失、点击数单位归一化、财富号正文失败统计等问题。

## 对比对象与正确口径

| 数据集 | 文件 | 角色 | 行数 | 时间范围 |
|---|---|---|---:|---|
| 项目原数据 | `数据/000002.csv` | 当前流水线的补爬基础 | `227,450` | `2011-07-06 00:00:00` 至 `2025-02-06 20:56:24` |
| 爬虫增强数据 | `data/000002_enhanced.csv` | 原数据 + 新增补爬结果 | `342,729` | `2011-07-06 00:00:00` 至 `2026-06-18 16:02:00` |
| 外部论文数据 | `outsource_check/000002.csv` | 高质量历史参考基准 | `208,996` | `2008-01-08 00:00:00` 至 `2018-12-31 20:47:00` |

关键事实：

- `数据/000002.csv` 与 `data/000002_enhanced.csv` 的 `post_id` 关系正常：项目原数据中的 `226,722` 个唯一 `post_id` 全部出现在 enhanced 文件中，没有丢失。
- enhanced 文件相对项目原数据新增 `115,279` 条，新增唯一 `post_id` 也是 `115,279` 个，说明新增部分没有重复。
- 新增记录全部晚于项目原数据最大时间 `2025-02-06 20:56:24`，符合“只补爬原数据之后的新帖”的流水线设计。
- `数据/000002.csv`、`data/000002_enhanced.csv` 与 `outsource_check/000002.csv` 按 `post_id` 求交集均为 `0`。这说明当前项目原数据与外部论文数据不是同一套历史底座。

所以，外部论文数据缺口应首先解释为 **历史基础数据源选择问题**，不是增量补爬阶段的直接质量失败。

## 历史基准数据的启示

外部论文数据覆盖 `2008-2018`，其中 2016 年尤其密集，有 `127,016` 条。当前项目原数据从 `2011` 开始，但在外部数据最核心的 `2016-2018` 年并没有与外部 `post_id` 对上。

这带来两个结论：

1. 如果研究目标需要复现实证论文或覆盖 2008-2018 历史段，`数据/000002.csv` 不是理想基础数据源。
2. 更合理的方案是把 `outsource_check/000002.csv` 作为新的历史底座，再用当前爬虫从 `2018-12-31 20:47` 之后做增量补爬。

注意：外部论文数据没有 `content` 字段，因此它更适合作为“帖子列表级”历史基准，而不是正文质量基准。若后续需要正文，可在外部数据基础上单独补正文，或接受普通帖正文等同标题的高速策略。

## 重点缺陷

### 1. 新增数据中 `post_type/user_id` 缺失较多

新增数据有 `24,065` 行 `post_type` 与 `user_id` 为空，占 `20.88%`。这些记录集中在 `2025-07` 到 `2025-12`，尤其：

| 月份 | 空 `post_type` 行数 |
|---|---:|
| 2025-11 | `12,300` |
| 2025-08 | `4,322` |
| 2025-10 | `3,888` |
| 2025-09 | `1,758` |
| 2025-12 | `1,403` |
| 2025-07 | `394` |

样例：

| post_id | 时间 | user_name | 标题 |
|---|---|---|---|
| 1632636810 | 2025-12-01 16:21 | 盆满万贯的原随云 | 必定回到1元，回到历史开始的地方，然后才能重生 |
| 1632636448 | 2025-12-01 16:19 | 账户已注销 | 最后再一个跌停侧底打破散户的信心 |
| 1632636374 | 2025-12-01 16:19 | 大唐鼎盛 | 万科的散户早就应该换金地和保利了... |

这些记录有标题、时间、URL、用户名，但缺少稳定用户 ID 和帖子类型。若研究只做按日发帖量或标题文本分析，还可使用；若做用户级行为、账号活跃度、帖子类型分层，则质量不足。

### 2. `click_count` 有 92 条非纯数字

新增数据中 `92` 条 `click_count` 为 `1.0万`、`3.9万`、`33.5万` 这类带中文单位的字符串，而外部论文数据该字段全为纯数字。

样例：

| post_id | 时间 | click_count | 标题 |
|---|---|---:|---|
| 20251201153807814387530 | 2025-12-01 15:38 | 3.9万 | 回到20年前！万科今日股价5.20... |
| 1632527915 | 2025-12-01 13:59 | 2.8万 | 资讯万科两只境内债券跌超30%... |
| 1630591427 | 2025-11-26 13:43 | 33.5万 | 资讯单日跌超30%！“22万科02”二次临停... |

建议在写 CSV 前统一把 `万` 转换为数字，例如 `3.9万 -> 39000`。

### 3. 财富号正文补爬质量不足

新增财富号 `post_type=20` 共 `3,103` 条，其中 `3,053` 条 `content` 为空。这个问题与当前速度优先策略有关：财富号 requests 失效后不再 Selenium 兜底，以避免大规模卡死。

如果后续研究只用标题，当前缺陷可接受；如果需要正文语义，财富号正文质量明显不够，需要单独补爬或标记排除。

### 4. 原数据自身存在重复 ID

enhanced 文件中仍有 `728` 个重复 `post_id`，涉及 `1,456` 行；这些重复来自项目原数据 `数据/000002.csv`，不是本次新增补爬制造的。

外部论文数据只有 `1` 个重复 ID。因此，如果要把项目数据做成高质量研究数据，Stage 3 最终仍应按 `post_id` 去重。

## 与外部论文数据相比，新增数据是否可用

按用途分层判断：

| 使用场景 | 判断 |
|---|---|
| 按日发帖量、标题文本、帖子 URL、评论数、转发数、基础时间序列 | 基本可用 |
| 只使用 `post_type` 非空的新增记录 | 列表字段质量接近外部论文数据 |
| 用户级分析、账号活跃度、用户网络 | 不完全可用，因为新增整体 `20.88%` 缺 `user_id` |
| 按帖子类型分层统计 | 不完全可用，因为新增整体 `20.88%` 缺 `post_type` |
| 点击数数值建模 | 需要先把 `万` 单位归一化 |
| 正文语义分析 | 普通帖可按当前策略使用标题；财富号正文缺失严重，不建议直接使用 |
| 历史研究复现 2008-2018 | 不能依赖当前项目原数据，应优先使用外部论文数据作为历史底座 |

所以，新爬数据的“列表级增量补爬”已经有可用价值，但整体还没有达到外部论文数据那种整洁程度。最影响比肩外部数据的不是新增重复或时间错误，而是 `post_type/user_id` 缺失与数值单位未归一。

## 建议的新流水线口径

### 1. 用外部数据作为更好的历史基础数据源

建议把 `outsource_check/000002.csv` 转换为当前流水线字段：

| 外部字段 | 当前字段 |
|---|---|
| `post_url` | `url` |
| `comment_count` | `coment_count` 或修正代码统一为 `comment_count` |
| `forward_count` | `forward` 或修正代码统一为 `forward_count` |
| 无 `content` | 可置空，或普通帖先用 `post_title` 填充 |
| 无 `post_source_id` | 置空 |

然后用转换后的外部数据作为 `000002` 的基础 CSV，再从 `2018-12-31 20:47` 之后增量补爬。这样才能同时拥有外部历史完整性和当前爬虫的新数据覆盖。

### 2. 修复新增记录 `post_type/user_id` 为空的问题

重点排查 2025 年 7-12 月列表解析路径。建议：

- 对 `post_type` 为空但 URL 形如 `https://guba.eastmoney.com/news,000002,{id}.html` 的普通帖，回填 `post_type=0`。
- 对资讯账号如 `万科A资讯`，若接口有类型字段，应保留真实类型；不要静默为空。
- 如果列表 HTML/API 当前拿不到 `user_id`，至少在质量报告中单独标记，避免用户级研究误用。

### 3. 归一化数值字段

写入 CSV 前统一处理：

- `1.0万 -> 10000`
- `3.9万 -> 39000`
- 空值或异常值保留为空并记录质量日志

### 4. Stage 3 最终去重

合并基础 CSV 与新增 CSV 后，按 `post_id` 去重。建议保留字段更完整的一条；字段完整度相同时保留较新的记录。

### 5. 财富号正文失败显式标注

继续保留 `{stock}_detail_failed.jsonl`，并在最终 enhanced CSV 或质量报告中统计：

- 财富号总数
- 财富号正文成功数
- 财富号正文失败数
- 失败原因样例

这样可以把“速度优先导致正文缺失”的取舍透明化。

## 最终判断

`data/000002_enhanced.csv` 的新增补爬部分在增量边界、去重、标题、时间、URL、评论数、转发数等列表级字段上表现不错，可以支撑一部分标题级和时间序列研究。

但它整体还不能完全比肩 `outsource_check/000002.csv` 的数据整洁度，主要短板是：

- 新增 `20.88%` 记录缺少 `post_type/user_id`。
- 新增 `92` 条点击数字段未数值化。
- 新增财富号正文空值率很高。
- 历史底座 `数据/000002.csv` 与外部论文数据完全不对齐。

更合理的下一步不是否定增量爬虫，而是：**用外部论文数据作为新的历史原数据集，再让当前高速流水线负责外部数据截止日期之后的新帖补爬；同时修复新增数据字段缺失和数值归一问题。**
