# 股票爬虫自动化流水线使用指南

> **本仓库仅包含代码，不包含历史基线数据。**
>
> 流水线支持两种模式：
> - **incremental（增量模式）**：需要历史 CSV，复现你之前的本地结果。
> - **full（全量模式）**：不需要历史 CSV，从网页按 `start_date` 全量爬取。
>
> 新需求默认使用 **full 模式**，从 `2009-01-01` 开始覆盖式全量爬取。

本文档从 `README.md` 的 2.4 自动化流水线说明转写而来，并补充 2000 支股票批量运行方式。

当前流水线以 CSV 为中心，不再把历史数据导入 MongoDB。默认目标是尽快稳定完成批量跑数：

- Stage 1：
  - incremental：读取历史 CSV，增量补爬新帖子。
  - full：从网页按 `start_date` 全量抓取帖子列表。
- Stage 2：补正文；普通股吧帖直接用标题填充，只有财富号正文走多线程 requests。
- Stage 3：
  - incremental：合并基础 CSV 与新帖子 CSV。
  - full：校验并导出 `full_posts.csv` 到 `data/{stock}_full_20090101.csv`。
- 不爬评论，不要求 MongoDB。

## 一、前置准备

### 1.1 代码与依赖

进入项目目录：

```powershell
cd e:\guba_project\EastMoney_Crawler
```

安装依赖：

```powershell
pip install -r requirements.txt
```

### 1.2 数据准备（必须）

**本项目要求为每只股票提供历史基线 CSV。** 原因：

- 东方财富股吧列表页只能按时间倒序访问，无法一次性拿到所有历史帖子。
- Stage 1 读取历史 CSV 后，以其最新日期为停止阈值，向前补爬新增帖子。
- 没有历史基线时，爬虫不知道从何处停止，且无法复现与本地一致的数据量。

请将历史 CSV 放在以下任一位置：

- 项目根目录：`EastMoney_Crawler\000001.csv`
- 历史数据目录：`EastMoney_Crawler\数据\000001.csv`
- 自定义目录：运行 Stage 1 或批量 worker 时用 `--source-dir` 指定

股票 CSV 文件名必须是 6 位股票代码，例如 `000001.csv`、`000002.csv`。

**CSV 字段要求：** 至少包含 `post_id`、`post_publish_time`、`post_title`、`url`、`content`，其余字段会自动补齐或留空。

### 1.3 验证数据是否就位

```powershell
python batch_worker.py --dry-run --limit 5 --source-dir e:\guba_project\EastMoney_Crawler\数据
```

输出应列出股票代码和对应 CSV 路径。若提示 `no stock CSV files discovered`，说明历史 CSV 没有放对位置。

## 二、单只股票三阶段运行

### Stage 1：提取历史数据 + 爬取帖子列表

```powershell
python auto_pipeline_000001.py --stock 000001 --stage 1 --source-dir e:\guba_project\EastMoney_Crawler\数据
```

> **必须显式传入 `--source-dir`**，否则 Stage 1 若找不到历史 CSV 会直接失败。不要把历史数据依赖藏在本地默认路径里。

该阶段会：

1. 优先从 `--source-dir\000001.csv` 读取历史 CSV。
2. 找不到时再查找项目根目录和默认 `数据` 目录。
3. 将基础 CSV 复制为 `temp_extract\000001_base.csv`。
4. 扫描历史 `post_id` 到内存 set，用于高速去重。
5. 从股吧列表页增量爬取新帖子，遇到旧日期自动停止。
6. 新帖子写入 `temp_extract\000001_new_posts.csv`。
7. 写入 `temp_extract\000001_stage1_manifest.json`，记录 source_rows / new_rows / source_sha256 / 源路径，供 Stage 3 校验。

如果你更新了历史 CSV 或想从头重建 base，加 `--force-refresh-base`：

```powershell
python auto_pipeline_000001.py --stock 000001 --stage 1 --source-dir e:\guba_project\EastMoney_Crawler\数据 --force-refresh-base
```

### Stage 2：爬取正文

```powershell
python auto_pipeline_000001.py --stock 000001 --stage 2 --detail-workers 3
```

该阶段会：

1. 筛选 `content` 为空的帖子。
2. 普通股吧帖不打开详情页，直接用 `post_title` 填充 `content`。
3. 财富号帖子用多线程 requests 补爬完整正文。
4. 财富号失效或空正文写入 `temp_extract\000001_detail_failed.jsonl`，后续重跑会跳过。
5. 正文更新先写增量 JSONL，最后统一写回 CSV。
6. 不爬评论，不需要 MongoDB。

`--detail-workers 1` 可用于保守回退，默认推荐 `3`。

### Stage 3：合并输出

```powershell
python auto_pipeline_000001.py --stock 000001 --stage 3
```

该阶段会：

1. **强校验**：读取 `temp_extract\000001_stage1_manifest.json`，确认 `*_base.csv` 行数与 manifest 一致。
2. 若 manifest 期望 `*_new_posts.csv` 有数据但该文件缺失，Stage 3 **会直接失败**，不再静默跳过。
3. 合并 `temp_extract\000001_base.csv` 与 `temp_extract\000001_new_posts.csv`。
4. 按 `post_publish_time` 降序排序。
5. 生成 `temp_export\000001_enhanced.csv`。
6. 校验 `out_rows == base_rows + new_rows`。
7. 当 `SKIP_BAIDU_UPLOAD=True` 时，复制到 `data\000001_enhanced.csv`。
8. 不导出评论 CSV。

如果 Stage 3 报错「manifest 缺失」或「base 行数与 manifest 不符」，请删除该股票的旧临时产物后重新跑 Stage 1：

```powershell
Remove-Item .\temp_extract\000001_*.csv -Force
Remove-Item .\temp_extract\000001_stage1_manifest.json -Force
Remove-Item .\.pipeline_flags\000001_*.done -Force
python auto_pipeline_000001.py --stock 000001 --stage 1 --source-dir e:\guba_project\EastMoney_Crawler\数据
```

## 三、批量 2000 股运行

> **警告：不要直接运行 `\batch_launcher.ps1` 而不带 `-SourceDir`。必须显式传入历史 CSV 目录，否则 worker 可能因找不到数据源而失败。**

批量模式使用：

- `batch_launcher.ps1`：一次启动多个 worker。
- `batch_worker.py`：自动发现股票 CSV、抢占任务、串行运行 Stage 1/2/3、写状态文件。
- `batch_progress\`：记录进度和失败原因。

每个 stock 在交给 worker 跑 Stage 1 之前，会自动清理自己的旧临时产物，避免复用过期 base/new。

推荐先做 dry-run：

```powershell
python batch_worker.py --dry-run --limit 5 --source-dir e:\guba_project\EastMoney_Crawler\数据
```

确认股票发现正常后，先用 2 个 worker 做 20 支试运行（所有命令必须带 `-SourceDir`）：

```powershell
.\batch_launcher.ps1 -WorkerCount 2 -DetailWorkers 3 -Limit 20 -SourceDir e:\guba_project\EastMoney_Crawler\数据
```

试运行通过后，推荐生产档位：

```powershell
.\batch_launcher.ps1 -WorkerCount 3 -DetailWorkers 3 -SourceDir e:\guba_project\EastMoney_Crawler\数据
```

如果内存不足，降为：

```powershell
.\batch_launcher.ps1 -WorkerCount 2 -DetailWorkers 2 -SourceDir e:\guba_project\EastMoney_Crawler\数据
```

如果 20 支试运行稳定、内存充足，再考虑：

```powershell
.\batch_launcher.ps1 -WorkerCount 4 -DetailWorkers 3 -SourceDir e:\guba_project\EastMoney_Crawler\数据
```

默认后台 worker 会隐藏运行，并把日志写入 `batch_logs\worker_*.log`。如果想看到独立窗口：

```powershell
.\batch_launcher.ps1 -WorkerCount 3 -DetailWorkers 3 -SourceDir e:\guba_project\EastMoney_Crawler\数据 -Visible
```

## 四、全量模式（full）：从 2009-01-01 起全量爬取

> **full 模式不需要历史 CSV。** 它以 `数据_list.csv` 为任务源，从网页抓取 `post_publish_time >= 2009-01-01` 的全部帖子。

### 单只股票 full 模式三阶段

```powershell
# Stage 1 full：从网页全量抓取帖子列表
python auto_pipeline_000001.py --stock 000001 --stage 1 --crawl-mode full --start-date 2009-01-01 --list-workers 6

# Stage 2 full：正文补爬（与增量模式相同）
python auto_pipeline_000001.py --stock 000001 --stage 2 --crawl-mode full --detail-workers 3

# Stage 3 full：校验并导出
python auto_pipeline_000001.py --stock 000001 --stage 3 --crawl-mode full
```

Stage 3 full 会执行以下校验：

1. `temp_extract\000001_full_manifest.json` 必须存在且 `crawl_mode=full`。
2. `failed_pages` 必须为空。
3. `temp_extract\000001_full_posts.csv` 行数与 manifest 一致。
4. `post_id` 无重复。
5. 最小日期不早于 `2009-01-01`。

任一校验失败都会直接报错，不会生成错误产物。

### 批量 full 模式运行

准备 `数据_list.csv`（单列股票代码），然后：

```powershell
# dry-run：确认能发现 4383 只股票
python batch_worker.py --crawl-mode full --stock-list .\数据_list.csv --dry-run --limit 10

# 小规模试跑（推荐先用 2 worker × 6 list-workers，跑 20 支）
.\batch_launcher.ps1 -WorkerCount 2 -DetailWorkers 3 -ListWorkers 6 -CrawlMode full -StartDate 2009-01-01 -Limit 20

# 生产运行（稳定后再提高并发）
.\batch_launcher.ps1 -WorkerCount 3 -DetailWorkers 3 -ListWorkers 6 -CrawlMode full -StartDate 2009-01-01
```

> **并发建议**：初始 `WorkerCount=3, ListWorkers=6`（总列表并发约 18）。若验证页、失败页增多，降为 `WorkerCount=2, ListWorkers=4`。不建议一上来就把 `WorkerCount * ListWorkers` 拉到 24 以上。

full 模式使用独立的进度目录：

```powershell
(Get-ChildItem batch_progress_full_20090101\*.done).Count
(Get-ChildItem batch_progress_full_20090101\*.failed).Count
```

输出文件名为 `data\{stock}_full_20090101.csv`，不会覆盖增量模式的 `data\{stock}_enhanced.csv`。

## 四、批量状态文件

### incremental 模式

每只股票在 `batch_progress\` 下使用这些状态文件：

| 文件 | 含义 |
|------|------|
| `{stock}.lock` | 某个 worker 正在处理该股票 |
| `{stock}.done` | Stage 1/2/3 已全部成功 |
| `{stock}.failed` | 该股票失败，需人工查看原因或后续补跑 |
| `{stock}.retrying` | 正在自动重试 |
| `{stock}.failed_upload` | Stage 3 上传失败，但本地产物已保留 |
| `{stock}.summary.json` | 耗时、输出大小、失败财富号数、退出码等摘要 |

### full 模式

full 模式使用 `batch_progress_full_20090101\` 下的同名状态文件，避免与增量任务混淆。

worker 使用原子 `.lock` 抢任务，避免多 worker 重复处理同一只股票。`.lock` 超过 3 小时未更新时，其他 worker 可回收。

## 五、监控与恢复

查看进度：

```powershell
(Get-ChildItem batch_progress\*.done).Count
(Get-ChildItem batch_progress\*.failed).Count
(Get-ChildItem batch_progress\*.failed_upload).Count
Get-ChildItem batch_progress\*.lock | Select-Object Name,LastWriteTime
```

查看某只股票摘要：

```powershell
Get-Content batch_progress\000001.summary.json
```

断电或重启后，直接重新启动 launcher：

```powershell
cd e:\guba_project\EastMoney_Crawler
.\batch_launcher.ps1 -WorkerCount 3 -DetailWorkers 3 -SourceDir e:\guba_project\EastMoney_Crawler\数据
```

worker 会自动跳过 `.done`，跳过默认 `.failed`，回收过期 `.lock`，继续处理剩余股票。

如果需要重新处理失败股票：

```powershell
.\batch_launcher.ps1 -WorkerCount 2 -DetailWorkers 2 -SourceDir e:\guba_project\EastMoney_Crawler\数据 -RetryFailed
```

## 六、磁盘策略

批量 worker 默认在每只股票开始前检查磁盘空间：

- 可用空间低于 `20GB`：暂停抢新任务，等待清理。
- 单只股票完成后，增强 CSV 默认保留在 `data\{stock}_enhanced.csv`。
- 全量 2000 支前，建议准备至少 `250GB` 可用空间，或确认百度网盘上传流程稳定后再开启上传清理。

建议先跑 20 支，统计 `data\` 目录增长速度，再决定本地留存还是网盘上传。

### incremental 全量前建议门槛

- 20 支试运行成功率 `>= 90%`。
- 平均每支耗时 `<= 30 分钟`。
- 没有重复处理同一只股票。
- `data\` 磁盘增长与预估一致。
- Stage 3 日志输出 `base_rows + new_rows = out_rows`。
- 验收行数：以你本地成功数据为基准，例如 `000002` 最终应接近 `342,729` 行。若明显偏低，请检查历史 CSV 是否放对、是否误删 `*_new_posts.csv`。

### full 全量前建议门槛

- 5 支单股票完整跑通 Stage 1 full → Stage 2 → Stage 3 full。
- Stage 1 full 日志中 `failed_pages` 为空。
- Stage 3 full 日志无校验错误。
- `data\{stock}_full_20090101.csv` 存在，且 `min(post_publish_time) >= 2009-01-01`。
- `post_id` 无重复。
- 50 支试跑成功率 `>= 90%`，平均每支耗时可控。

### 如何排查别人复现结果偏少（incremental）

1. 检查是否传了 `-SourceDir`，以及该目录下是否有对应股票 CSV。
2. 检查 `temp_extract\{stock}_stage1_manifest.json` 是否存在、`new_rows` 是否 > 0。
3. 检查 Stage 3 日志是否有「manifest 缺失」「new_posts 行数不符」「整合后行数异常」等错误。
4. 若曾用错误产物当基线，先删除该股票全部临时状态再重跑：

```powershell
Remove-Item .\temp_extract\000002_*.csv -Force
Remove-Item .\temp_extract\000002_stage1_manifest.json -Force
Remove-Item .\.pipeline_flags\000002_*.done -Force
Remove-Item .\batch_progress\000002.* -Force
python auto_pipeline_000001.py --stock 000002 --stage 1 --source-dir e:\guba_project\EastMoney_Crawler\数据
python auto_pipeline_000001.py --stock 000002 --stage 2
python auto_pipeline_000001.py --stock 000002 --stage 3
```

### 如何排查 full 模式失败

1. 检查 `batch_progress_full_20090101\{stock}.failed` 中的 `failed_reason`。
2. 检查 `temp_extract\{stock}_full_manifest.json`：
   - `failed_pages` 是否为空；
   - `rows` 是否与 CSV 实际行数一致；
   - `min_time` 是否 >= `2009-01-01`。
3. 若某只股票失败，清理后重跑：

```powershell
Remove-Item .\temp_extract\000001_full_posts.csv -Force
Remove-Item .\temp_extract\000001_full_manifest.json -Force
Remove-Item .\.pipeline_flags\000001_*.done -Force
Remove-Item .\batch_progress_full_20090101\000001.* -Force
python auto_pipeline_000001.py --stock 000001 --stage 1 --crawl-mode full --start-date 2009-01-01 --list-workers 6
python auto_pipeline_000001.py --stock 000001 --stage 2 --crawl-mode full
python auto_pipeline_000001.py --stock 000001 --stage 3 --crawl-mode full
```
