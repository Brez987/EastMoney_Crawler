## 二、已发现的问题清单

### 问题 1：Windows 文件锁定冲突（PermissionError）⚠️ 高危

**现象**：
```
PermissionError: [WinError 32] 另一个程序正在使用此文件，进程无法访问。
  → batch_worker.py: write_json() → os.replace(tmp_path, path)
  → auto_pipeline_000001.py: update_csv_content() → shutil.move(tmp_path, csv_path)
```

**触发条件**：
- 多个 Worker 同时写入同一只股票的进度文件（`.failed`、`.retrying`）
- CSV 文件被上一个 stage 的进程持有，`shutil.move` 时冲突
- 反病毒软件或 Windows 索引服务短暂锁定文件

**影响**：Worker 崩溃退出，该股票标记为失败（即使数据已完整抓取）

**已修复** ✅：
- `batch_worker.py` → `write_json()` 添加 5 次重试（递增延迟 0.5s~2.5s）
- `auto_pipeline_000001.py` → 新增 `_safe_replace()` 函数，所有 `os.replace`/`shutil.move` 走重试逻辑
- 但仍需注意：重试无法解决持续性的文件锁定（如反病毒软件持有）

**建议后续改进**：
- 为每个股票使用唯一临时文件名（加 PID 后缀）
- `update_csv_content` 在重试前先 `time.sleep(1)` 等待文件释放

### 问题 2：UnicodeDecodeError — 财富号返回非 UTF-8 数据 ⚠️ 中危

**现象**：
```
UnicodeDecodeError: 'utf-8' codec can't decode bytes in position 0-1: unexpected end of data
[000567][stage 2] exit=1, seconds=1.6
```

**触发条件**：
- 财富号详情页返回二进制/garbled 数据（可能是反爬页面、图片验证码或 gzip 压缩异常）
- curl_cffi 请求返回的 `response.content` 无法以 UTF-8 解码

**影响**：Stage 2 立即退出，股票标记失败（3 次重试全部失败）

**未修复** ❌：代码中缺少对 `response.content` 的编码容错处理

**建议修复方案**：
```python
# auto_pipeline_000001.py 财富号爬取部分
try:
    text = response.content.decode('utf-8')
except UnicodeDecodeError:
    try:
        text = response.content.decode('gbk')  # 尝试 GBK
    except UnicodeDecodeError:
        text = response.content.decode('utf-8', errors='replace')  # 容错
```

### 问题 3：Worker 静默退出（无自动恢复）⚠️ 高危

**现象**：
- Worker 日志突然中断（进度显示到 75% 后无后续输出）
- Python 进程数从 6 个逐步降到 0 个
- 无任何错误日志（stdout/stderr 均为空）

**触发条件**：
- 子进程（auto_pipeline）遇到未捕获异常直接 `exit(1)`
- batch_worker 的 `process_claim()` 收到非零退出码后，重试耗尽则标记失败
- 当 `find_next_claim()` 返回 None 时，worker 正常退出（但实际还有数千只股票待处理）

**根因分析**：
- `find_next_claim()` 遍历股票列表时，如果所有"下一个"股票都被其他 Worker 的锁文件占住，返回 None
- 当只剩 1 个 Worker 时，如果它刚处理完一只股票，下一只被已死 Worker 的残留锁文件阻塞 → 退出
- **整个系统没有守护进程/监控/自动重启机制**

**未修复** ❌

**建议修复方案**：
- `find_next_claim()` 返回 None 前等待 30-60 秒并重试，而非立即退出
- 在 `worker_loop` 外层加 try/except 捕获所有异常并继续
- 添加外部监控脚本（每 5 分钟检查 Python 进程数，为 0 则重启 launcher）

### 问题 4：Python stdout 缓冲导致日志不可见 ⚠️ 低危

**现象**：
- `batch_logs/worker_*.out.log` 文件大小为 0 或长期不更新
- Worker 实际在跑（CPU 活跃、temp_extract 在写入），但日志无输出

**触发条件**：
- Windows 上 Python 默认使用块缓冲（非行缓冲）
- 通过 `Start-Process -RedirectStandardOutput` 重定向时，缓冲更加严重
- 仅在进程退出或缓冲区满（通常 8KB）时才 flush

**影响**：无法通过日志判断 Worker 是否卡死或在正常运行，导致误判（如本次会话中多次误杀）

**未修复** ❌

**建议修复方案**：
```powershell
# batch_launcher.ps1 — 设置环境变量禁用缓冲
$env:PYTHONUNBUFFERED = "1"
```
```python
# batch_worker.py 开头
sys.stdout.reconfigure(line_buffering=True)  # Python 3.7+
```

### 问题 6：多 Worker 竞争同一股票 ⚠️ 中危

**现象**：
- `batch_progress_full_20090101/000004.done` 和 `000004.failed` 同时存在
- 同一只股票被两个 Worker 先后处理

**触发条件**：
- Worker A 认领股票后崩溃，锁文件残留
- Worker B 的 `claim_stock()` 中的 `is_stale()` 判定锁过期（默认 3 小时），回收锁
- 但 Worker A 可能后来恢复了（如文件锁释放后重试成功），导致两者都写入状态

**影响**：股票被重复爬取，浪费资源；`.done` 和 `.failed` 共存导致后续 Worker 行为不确定

**部分缓解**：`.done` 优先级高于 `.failed`（`claim_stock` 先检查 `.done`）

**建议修复方案**：
- 缩短 `stale-lock-hours` 默认值（3h → 1h）
- 在 `write_json` 时检查目标文件是否已被其他 Worker 写入（比较 worker_id）

### 问题 8：零磁盘预警机制 ⚠️ 低危

**现象**：当前磁盘消耗速度约 0.15 GB/只，按 5,565 只计算需约 **835 GB**。当前仅剩 **107 GB**。

**现有机制**：`batch_worker.py` 在认领新股票前检查 `--min-free-gb`（默认 20 GB），低于阈值则等待

**不足**：检查仅基于瞬时剩余空间，未做趋势预测。当剩余空间不足以完成下一只股票的完整三阶段时，会在中途失败（浪费已爬数据）。

**建议修复方案**：
- 在 Stage 1 启动前估算所需空间（页面数 × 平均帖子大小）
- 当前剩余空间 < 估算所需空间 × 2 时暂停认领新股票
